#SQL 
# 问题

在一次数据开发中，想实现查询每天峰值时刻多维分量的需求。这个需求简单描述如下：

有一张记录了网络流量的数据表 netflow，包含 Scene、AggTime、Provider、Region 等字段。现在需要计算网络峰值时候的多维分量。就是首先要找到整体流量的最大值的时间，然后将这个时间点上的所有流量数据。但是这里需要两种流量的最大值，一种的全局的，另一种是每个 scene 下的。

需求比较清晰。一种思路是从 netflow 表种分别实现这两种数据，然后通过一个 hive 任务合并成最终的一个结果表，实现如下(可以不看)：
```sql
-- by全局
with total_max as (
	select AggTime as AggTime, Sum(TotalBytes) as max_v
    from netflow
    where toDate(AggTime) = today()-1
    group by AggTime
    order by max_v desc
    limit 1
  ) select SumMerge(a.TotalBytes) as v,a.AggTime as t,a.Provider, a.Region,a.Scene
  from netflow as a
  inner join total_max as b on a.AggTime = b.AggTime
  where toDate(a.AggTime) = today()-1
  group by a.AggTime,  a.Provider ,a.Region,a.Scene
  order by v desc
)

-- by scene
with scene_max as (
SELECT 
    argMax(AggTime, v) AS AggTime,
    Scene,
    max(v) AS max_v
FROM (
    SELECT 
        AggTime,
        Scene,
        SumMerge(TotalBytes) AS v
    FROM netflow
    WHERE toDate(AggTime) = today() - 1
    GROUP BY AggTime, Scene
)
GROUP BY Scene
ORDER BY max_v DESC
) select SumMerge(a.TotalBytes) as v, a.AggTime as t, a.Provider, a.Region, a.Scene
from netflow as a
inner join scene_max as b on b.Scene = a.Scene and b.AggTime = a.AggTime
where toDate(a.AggTime) = today() - 1
group by a.AggTime, a.Provider, a.Region, a.Scene
order by v desc
```

到目前为止都没有什么问题。接下来合并两张表就行了：
```sql
SELECT
	COALESCE(total.dt, scene.dt) AS dt,
	COALESCE(total.provider, scene.provider) AS provider,
	COALESCE(total.region, scene.region) AS region,
	COALESCE(total.scene, scene.scene) AS scene,
	COALESCE(total.total_bytes, 0) AS max_total_bits,
	total.agg_time AS max_total_bits_time,
	COALESCE(scene.total_bytes, 0) AS max_scene_bits,
	scene.agg_time AS max_scene_bits_time
FROM
	total_max AS total
	FULL OUTER JOIN
	scene_max AS scene
ON 
	total.dt = scene.dt
	AND total.scene = scene.scene
	AND total.provider = scene.provider
	AND total.region = scene.region
WHERE
	total.dtm = '20250220'
	AND scene.dtm = '20250220'

```

 稍微解释一下这个 SQL，这里实际存在一个在线 CK转离线 HIVE的过程，所以colum name 都变成小写，同时需要过滤掉 hive 的分区，不过这并不重要。到这就出现了问题，完成之后一步之后，再验证，发现数据少了，结果和两张过程表相比，少了数据。
 
 排查的过程比较辛酸(让 AI 帮忙分析也没有效果)，但思路其实很简单：选择一个小数据集的条件，用这个条件同时对三个表进行查询，手动检查少了那部分数据。发现在这个条件下，total表有 6 条数据，scene 表有 1100条数据，而结果表却只有 6 条数据。根据验证的结果，SQL执行的表现从 FULL OUTER JOIN 退化成了 INNER JOIN。

# 原因

这里直接说原因：合并 SQL的执行顺序是先执行 join，再执行 where。 这里出现这个问题就是因为total 表和 scene 表 join 中，因为大量数据都是不匹配的(1100 vs 6)，产生了大量的 NULL 填充。join 之后 `total.dtm = '20250220'` 这个条件将 NULL值都过滤了，只留下了匹配上的 6 条。

# 排查&验证

这个问题排查和验证的思路相同。理论上两个表分别为 6 和 1100 条，在 FULL OUTER JOIN 之后，理论上至少应该有 1100条。现在只出现了 6 条，说明数据被过滤了。问题出现在 on 或者 where 上。

我们通过 with 将 where 取消，只保留 on：
```sql
with total_max as (select * from total_max where dtm = 'xxx'),
scene_max as (select * from scene_max where dtm = 'xxx')
SELECT
	COALESCE(total.dt, scene.dt) AS dt,
	COALESCE(total.provider, scene.provider) AS provider,
	COALESCE(total.region, scene.region) AS region,
	COALESCE(total.scene, scene.scene) AS scene,
	COALESCE(total.total_bytes, 0) AS max_total_bits,
	total.agg_time AS max_total_bits_time,
	COALESCE(scene.total_bytes, 0) AS max_scene_bits,
	scene.agg_time AS max_scene_bits_time
FROM
	total_max AS total
	FULL OUTER JOIN
	scene_max AS scene
ON 
	total.dt = scene.dt
	AND total.scene = scene.scene
	AND total.provider = scene.provider
	AND total.region = scene.region
```

查询结果1100 条，符合期望。至此，验证改问题是 where 的导致。

# 扩展

## SQL的标准执行顺序

通过这个小问题，可以在扩展一下 sql 的执行逻辑以及什么情况下出现 OUTER JOIN 退化成 INNER JOIN。

SQL 的执行顺序如下所示：
1. **WITH**：
    - 定义并执行 CTE，生成临时结果集。
2. **FROM**：
	- 确定查询的源表，包括表、视图或子查询。
    - 如果涉及多表连接，生成笛卡尔积。
3. **ON** :
    - 对连接条件进行过滤（仅适用于 `JOIN` 操作）。
    - 例如，`LEFT JOIN` 中 `ON` 条件会过滤右表，但保留左表所有行。
4. **JOIN** :
    - 根据 `ON` 条件，将符合条件的行进行连接（如 `INNER JOIN`、`LEFT JOIN` 等）。
5. **WHERE**：
    - 对连接后的结果集进行过滤。
    - 注意：`WHERE` 会过滤掉不符合条件的行，可能导致 `LEFT JOIN` 的左表数据丢失。
6. **GROUP BY**：
    - 将数据按指定列分组，通常用于聚合操作（如 `SUM`、`COUNT` 等）。
7. **HAVING**：
    - 对分组后的结果集进行过滤（类似于 `WHERE`，但用于聚合后的数据）。
8. **SELECT**：
    - 选择要返回的列，包括计算列、聚合函数等。
    - 注意：`SELECT` 中的别名在此阶段生成，不能在 `WHERE` 或 `GROUP BY` 中使用，只能在 `ORDER BY` 中使用。(每个 SQL引擎的行为不一样，例如 MYSQL 可以在 HAVING 中使用SELECT别名)。
9. **DISTINCT**：
    - 对 `SELECT` 的结果集去重。
10. **ORDER BY**：
    - 对最终结果集进行排序。
    - 可以使用 `SELECT` 中定义的别名。
11. **LIMIT / OFFSET**：
    - 限制返回的行数（如分页查询）。

参考如下的 SQL：
```sql
SELECT 
    department, 
    COUNT(*) AS employee_count
FROM 
    employees
JOIN 
    departments ON employees.dept_id = departments.id
WHERE 
    employees.salary > 5000
GROUP BY 
    department
HAVING 
    COUNT(*) > 10
ORDER BY 
    employee_count DESC
LIMIT 5;
```

这条 SQL的标准执行顺序应该为先通过 FROM 找到对应的库表，然后根据 ON 的条件进行 JOIN，然后对上一步结果集进行 WHERE 的过滤，进一步减少结果集。接下对结果集进行 GROUP ，在 GROUP 的基础上 HAVING 过滤。然后才执行 SELECT 中的表达式，包括聚合函数等等。最后在进行 ORDER 和 LIMIT。

这里稍废篇章对 SQL的执行顺序做了一些描述，但这是理论上的。实际上数据库优化器会根据索引、统计信息等调整实际执行顺序，例如： `WHERE` 条件提前到 `JOIN` 前执行。  

```sql
with total_max as ( 
select 
	AggTime as AggTime, 
	SumMerge(TotalBytes) as max_v 
from netflow
where toDate(AggTime) = today()-1 
group by AggTime 
order by max_v desc 
limit 1 
) select 
	SumMerge(a.TotalBytes) as v, 
	a.AggTime as t, 
	a.Provider,  
	a.Region,
	a.Scene 
from rnp.dm_flowlog_flink_sink_minute_agg as a 
inner join total_max as b 
on 
	a.AggTime = b.AggTime 
	and toDate(a.AggTime) = today()-1 
-- where toDate(a.AggTime) = today()-1 
group by 
	a.AggTime, 
	a.Provider, 
	a.Region, 
	a.Scene 
order by v desc
```

 在上面这 SQL中，环境为 CK， AggTime 是第一排序键，将toDate(a.AggTime) = today()-1放在WHERE 中会比放在 ON 中快不少。这里优化器发现 `WHERE` 中的条件能大幅减少数据量，优先执行 `WHERE` 过滤，直接减少主表 `a` 的数据量，再进行连接。而 `ON` 中的条件可能被优化器视为连接逻辑的一部分，无法提前过滤，会增加连接时的数据量（尤其是当 `a.AggTime` 包含大量不符合条件的数据时）。

这里在多费一些笔墨讨论一下连接和过滤。假设A有 50 条数据，B有 50 条。 `A join B on A.id > 20` 会导致A 拿 50 条，B拿 50 条去join。而 `A join B where A.id > 20` 受优化器的影响，最终 A拿 30 条，B拿50 条 join。 而 on 是 join 过程中的条件。

 如果 `AggTime` 列有索引，`WHERE` 条件可以利用索引快速过滤数据。而 `ON` 中的条件可能无法有效利用索引，导致全表扫描或更慢的过滤。因此我们也可以得到一个最佳实践，连接条件中有索引，可以放到 where 中利用索引剪集。

## 还有那些情况会导致OUTER JOIN退化为 INNER JOIN

`OUTER JOIN` 退化为 `INNER JOIN` 的主要原因是通过 `WHERE` 条件、`ON` 子句的非 `NULL` 约束或联合过滤，导致未匹配的行被意外排除。解决方法是合理设计过滤条件的放置位置（如将过滤提前到子查询或移到 `ON` 子句中）。

常见的场景有：

### 1. 在 WHERE 子句中对被驱动表的非 NULL 列进行过滤

```sql
-- 若 `b.column` 为 `NULL`（在未匹配的行中），`WHERE` 条件 `b.column = 'value'` 会排除这些行，导致结果与 `INNER JOIN` 相同。
SELECT *
FROM table_a a
LEFT JOIN table_b b ON a.id = b.a_id
WHERE b.column = 'value';  -- 过滤被驱动表非 NULL 的列
```

- 问题：当在 `OUTER JOIN` 后，对「被驱动表（右表）」的非 NULL 列使用 `WHERE` 条件过滤时，会强制排除驱动表中「未匹配到的行」（因为这些行的被驱动表字段为 `NULL`，无法满足条件）。 
- 解决方法：将被驱动表的过滤条件移到 `ON` 子句中：
```sql
SELECT *
FROM table_a a
LEFT JOIN table_b b 
  ON a.id = b.a_id AND b.column = 'value';  -- 过滤条件放在 ON 中
```

### 2. JOIN 条件中强制要求被驱动表的字段非 NULL

```sql
-- 若 `b.column` 必须非 `NULL`，则未匹配的行无法满足此条件，最终结果会丢失这些行。
SELECT *
FROM table_a a
FULL OUTER JOIN table_b b 
  ON a.id = b.a_id AND b.column IS NOT NULL;  -- 要求被驱动表字段非 NULL
```
- 问题：  在 `JOIN` 的 `ON` 子句中，如果要求被驱动表的字段必须满足某个非 `NULL` 条件，则未匹配的行会被自动排除。  

### 3. 在 FULL OUTER JOIN 后同时过滤两个表的字段

```sql
-- 如果某行在 `table_a` 或 `table_b` 中不存在（即未匹配），则 `a.dtm` 或 `b.dtm` 为 `NULL`，无法通过 `WHERE` 条件，导致结果等同于 `INNER JOIN`。
SELECT *
FROM table_a a
FULL OUTER JOIN table_b b ON a.id = b.a_id
WHERE a.dtm = '20250101' AND b.dtm = '20250101';
```

- 问题：对 `FULL OUTER JOIN` 后的结果，在 `WHERE` 子句中同时过滤两个表的字段（尤其是通过 `AND` 连接），会导致只有「完全匹配的行」被保留。  
- 解决方法：提前在子查询中分别过滤两个表，再进行 `FULL OUTER JOIN`：
```sql
WITH 
filtered_a AS (SELECT * FROM table_a WHERE dtm = '20250101'),
filtered_b AS (SELECT * FROM table_b WHERE dtm = '20250101')
SELECT *
FROM filtered_a a
FULL OUTER JOIN filtered_b b ON a.id = b.a_id;
```

### 4. 数据库优化器的隐式转换

```sql
-- 此处 `WHERE b.a_id IS NOT NULL` 直接排除所有未匹配的行，结果与 `INNER JOIN` 相同。
SELECT *
FROM table_a a
LEFT JOIN table_b b ON a.id = b.a_id
WHERE b.a_id IS NOT NULL;  -- 隐含要求「必须匹配」
```

- 问题：某些数据库优化器可能发现 `WHERE` 条件或被驱动表过滤条件隐含了「必须匹配」，从而自动将 `OUTER JOIN` 优化为 `INNER JOIN`。
